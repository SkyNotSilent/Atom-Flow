# AtomFlow 架构设计文档

## 系统概览

AtomFlow 是一个基于 AI Agent 的知识工作台，通过三个协同工作的 Agent 实现「信息消费 → 知识沉淀 → 内容创作」的完整闭环。

---

## 核心设计理念

### 1. 知识原子化（Atomization）

传统的知识管理工具以「文章」为单位存储，导致：
- 检索粒度太粗：搜索时返回整篇文章，需要人工定位有用片段
- 复用成本高：想引用某个数据或观点，需要重新阅读全文
- 关联难发现：不同文章中的相关观点无法自动关联

**AtomFlow 的解法**：将文章拆解为「原子卡片」（观点/数据/金句/故事），每张卡片是最小的可复用知识单元。

### 2. Agent 协作架构

不同于单一 AI 助手，AtomFlow 设计了三个专职 Agent：
- **Agent 1（知识原子化）**：专注内容理解与拆解
- **Agent 2（写作助手）**：专注素材召回与生成
- **Agent 3（风格定制）**：专注风格理解与指令生成

每个 Agent 只做一件事，通过数据库协作，避免单一 Agent 的能力边界模糊。

## Agent 详细设计

### Agent 1: 知识原子化 Agent

#### 输入
- 文章 URL 或全文内容
- 用户触发：点击「原子化存入知识库」

#### 处理流程

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 全文提取                                                   │
│    • 使用 Readability 提取正文                                │
│    • 使用 JSDOM 解析 HTML 结构                                │
│    • 清理广告、导航栏等噪音                                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 2. AI 语义分析                                                │
│    • 识别文章主题与结构                                        │
│    • 定位关键段落（论点、数据、案例）                           │
│    • 理解上下文依赖关系                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 3. 智能拆分为四类卡片                                          │
│    • 观点卡片：核心论点、判断、结论                             │
│    • 数据卡片：数字、统计、研究结果                             │
│    • 金句卡片：可直接引用的精彩表达                             │
│    • 故事卡片：案例、场景、叙事片段                             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 4. 持久化存储                                                 │
│    • saved_articles 表：保存原文                              │
│    • saved_cards 表：保存卡片，关联 saved_article_id          │
│    • 记录卡片来源（origin: 'ai' / 'manual'）                  │
└─────────────────────────────────────────────────────────────┘
```

#### 输出
- 4-12 张原子卡片（根据文章长度）
- 每张卡片包含：
  - `content`：卡片内容
  - `card_type`：观点/数据/金句/故事
  - `saved_article_id`：原文 ID（可回溯）
  - `origin`：ai（AI 生成）或 manual（用户手动创建）

#### 设计亮点

**语义理解 vs 规则切分**
- ❌ 简单方案：按段落切分，每段一张卡片
- ✅ AtomFlow：理解内容类型，一个论点可能跨越多段，一段可能包含多个数据点

**上下文保留**
- 每张卡片记录 `saved_article_id`，点击可跳转原文
- 避免「孤立卡片」问题：看到一个数据，不知道来自哪篇文章、什么背景

**去重机制**
- 相似卡片自动合并（未来基于向量相似度）
- 避免知识库冗余：同一个观点被多篇文章提及时，只保留一张卡片，记录多个来源

### Agent 2: 写作助手 Agent

#### 输入
- 用户的写作请求（如「写一篇关于知识管理的文章」）
- 对话历史（支持多轮交互）
- 用户选择的写作风格 Skill

#### LangGraph 状态图

```
START
  ↓
┌─────────────────────────────────────────────────────────────┐
│ hydrate_context                                              │
│ • 加载用户信息（user_id, nickname）                           │
│ • 加载对话历史（最近 10 条消息）                               │
│ • 初始化状态变量                                              │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ load_effective_skills                                        │
│ • 从 write_style_skills 表读取用户选择的 Skill                │
│ • 合并系统基线 Skill（如「不编造来源」）                       │
│ • 格式化为 AI 可理解的指令                                     │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ classify_intent                                              │
│ • 分类用户意图：                                              │
│   - write_article：写一篇完整文章                             │
│   - answer_question：回答一个问题                             │
│   - revise_content：修改已有内容                              │
│   - clarify：需要追问澄清                                     │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ retrieve_knowledge                                           │
│ • 从用户消息中提取关键词                                       │
│ • 在 saved_cards 表中检索相关卡片                             │
│ • 当前：关键词匹配（ILIKE）                                   │
│ • 未来：向量语义搜索（pgvector）                              │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ enrich_sources                                               │
│ • 对于检索到的卡片，补充原文语境                               │
│ • 从 saved_articles 表读取原文标题、URL                       │
│ • 构建完整的引用信息                                          │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ decide_next                                                  │
│ • 决策：是否有足够信息生成内容？                               │
│   - 如果 intent = clarify → 追问用户                          │
│   - 如果检索到相关卡片 → 生成内容                             │
│   - 如果无相关卡片 → 提示用户补充知识库                        │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ generate_answer_or_draft                                     │
│ • 调用 LLM 生成内容                                           │
│ • 注入上下文：                                                │
│   - 用户的写作风格 Skill                                      │
│   - 检索到的知识卡片                                          │
│   - 对话历史                                                 │
│ • 流式输出（SSE）                                             │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ persist_memory                                               │
│ • 保存本轮对话到数据库（未来功能）                             │
│ • 更新用户偏好（如常用主题、风格）                             │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ respond                                                      │
│ • 返回生成的内容给前端                                         │
│ • 附带元数据：使用的卡片、引用来源                             │
└─────────────────────────────────────────────────────────────┘
  ↓
END
```

#### 状态变量（LangGraph Annotation）

```typescript
{
  userId: number;                    // 用户 ID
  threadId: string;                  // 对话线程 ID
  userMessage: string;               // 用户当前消息
  conversationHistory: Message[];    // 对话历史
  effectiveSkills: Skill[];          // 生效的写作风格
  intent: "write_article" | "answer_question" | "revise_content" | "clarify";
  retrievedCards: Card[];            // 检索到的知识卡片
  enrichedSources: Source[];         // 补充了原文信息的来源
  shouldGenerate: boolean;           // 是否应该生成内容
  generatedContent: string;          // 生成的内容
  graphTrace: TraceItem[];           // 执行轨迹（用于调试）
}
```

#### 设计亮点

**意图分类的价值**
- 「写一篇文章」→ 需要检索多张卡片，生成长文
- 「这段话怎么改」→ 不需要检索，直接修改
- 「解释一个概念」→ 检索相关卡片，生成简短回答

不同意图走不同路径，避免无效检索和生成。

**知识召回策略**
- 当前：关键词匹配（`WHERE content ILIKE '%关键词%'`）
- 未来：向量语义搜索（`ORDER BY embedding <=> query_embedding LIMIT 10`）

关键词匹配的问题：「知识管理」无法召回「个人笔记系统」相关卡片（语义相关但词汇不同）。

**原文引用机制**
- 生成内容时，AI 会标注使用了哪些卡片
- 前端展示时，卡片可点击跳转原文
- 避免「AI 幻觉」：所有论据都有出处

### Agent 3: 风格定制 Agent

#### 输入
- 用户的风格描述（如「像产品经理面试复盘，必须讲机制和取舍」）
- 可选：样本文本（用户粘贴一段文字，AI 分析其风格）

#### LangGraph 状态图

```
START
  ↓
┌─────────────────────────────────────────────────────────────┐
│ analyze_user_input                                           │
│ • 判断输入类型：                                              │
│   - description：纯文字描述风格                               │
│   - sample：纯样本文本                                        │
│   - both：描述 + 样本                                         │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ extract_style_features                                       │
│ • AI 分析并提取：                                             │
│   - tone（语气）：专业 / 轻松 / 严谨 / 幽默                    │
│   - structure（结构）：总分总 / 场景开头 / 数据驱动            │
│   - citationStyle（引用风格）：学术引用 / 口语化 / 不引用      │
│   - constraints（约束）：必须遵守的规则                        │
│   - examples（示例）：具体的表达示例                           │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ generate_skill_draft                                         │
│ • AI 生成结构化 Skill 定义：                                  │
│   {                                                          │
│     name: "产品经理面试体",                                   │
│     description: "适合产品经理面试复盘场景...",                │
│     prompt: "写作时遵循以下规则：1. 必须讲机制...",            │
│     constraints: ["不用空泛排比", "每个观点讲取舍"],           │
│     examples: ["示例1", "示例2"]                              │
│   }                                                          │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐
│ validate_and_format                                          │
│ • 校验字段长度（name ≤ 40 字符，description ≤ 180 字符）      │

│ • 去重和格式化                                                │
└─────────────────────────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────────────────────────┐

│ • 返回草案给前端                                              │
│ • 用户确认后，前端调用 POST /api/write/agent/skills 保存      │
└─────────────────────────────────────────────────────────────┘
  ↓

```

#### 输出
- 结构化的 Skill 定义（JSON）
- 保存到 `write_style_skills` 表
- Agent 2 在 `load_effective_skills` 节点读取并应用

#### 设计亮点

**自然语言 → 结构化指令**

用户输入：
```
像产品经理面试复盘，必须讲机制和取舍，不要空泛排比
```

AI 生成：
```json
{
  "name": "产品经理面试体",
  "description": "适合产品经理面试复盘场景，强调机制设计与取舍分析",
  "prompt": "写作时遵循以下规则：\n1. 每个功能或决策必须解释背后的机制\n2. 明确说明取舍：选择 A 意味着放弃 B\n3. 避免使用排比句和空泛形容词\n4. 用具体案例支撑观点",
  "constraints": [
    "不使用排比句",
    "每个观点必须讲机制和取舍",
    "避免空泛形容词（如'非常好'、'很重要'）"
  ],
  "examples": [
    "我们选择了异步处理，牺牲了实时性，换取了系统稳定性。",
    "这个功能的核心机制是：用户每次操作触发一次计算，而非全量重算。"
  ]
}
```

**样本分析能力**

用户粘贴一段文字：
```
AI 产品的核心不是技术，而是用户体验。我们在设计 AtomFlow 时，首先明确了三个目标：降低知识管理门槛、提升内容复用率、保持原文语境。

为了实现这些目标，我们设计了原子卡片机制：每篇文章自动拆解成观点、数据、金句、故事四类卡片。这个机制的关键取舍是：牺牲了完整性，换取了可组合性。
```

AI 分析后生成：
```json
{
  "name": "产品设计复盘体",
  "tone": ["理性", "结构化", "目标导向"],
  "structure": ["先明确目标", "再讲机制", "最后讲取舍"],
  "constraints": [
    "每个设计决策必须说明目标",
    "必须讲清楚机制（如何实现）",
    "必须讲清楚取舍（牺牲了什么）"
  ]
}
```

### 典型用户旅程

**场景：用户想写一篇关于「AI 产品设计」的文章**

1. **信息消费阶段**（过去一周）
   - 用户在「今日推送」看到 5 篇相关文章
   - 每篇点击「原子化存入知识库」
   - **Agent 1** 拆解为 30 张卡片（观点 12 张、数据 8 张、金句 6 张、故事 4 张）

2. **风格定制阶段**（写作前）
   - 用户在「写作页」点击「Skills 助手」
   - 输入：「像产品经理面试复盘，必须讲机制和取舍，不要空泛排比」
   - **Agent 3** 生成「产品经理面试体」Skill
   - 用户确认并保存

3. **内容创作阶段**（写作时）
   - 用户在「写作页」输入：「写一篇关于 AI 产品设计的文章，重点讲用户体验」
   - **Agent 2** 执行：
     - `classify_intent` → 识别为 `write_article`
     - `retrieve_knowledge` → 检索到 12 张相关卡片（包含「用户体验」关键词）
     - `enrich_sources` → 补充原文信息（文章标题、URL）
     - `load_effective_skills` → 加载「产品经理面试体」Skill
     - `generate_answer_or_draft` → 生成文章草稿，应用风格约束
   - 用户看到草稿，点击卡片引用可跳转原文

## 技术选型理由

### 为什么选择 LangGraph？

**对比其他方案**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **简单 Prompt Chain** | 实现简单 | 无状态管理、无条件分支 | 单步任务（如翻译） |
| **LangChain LCEL** | 链式调用、可组合 | 复杂逻辑难以表达 | 线性流程（如 RAG） |
| **LangGraph** | 状态图、条件分支、循环 | 学习曲线陡峭 | 多步骤 Agent |
| **自研状态机** | 完全可控 | 开发成本高、难维护 | 特殊需求 |

**AtomFlow 选择 LangGraph 的原因**：
- Agent 2 需要条件分支（根据意图决定是否检索知识库）
- 需要状态管理（对话历史、检索结果、生效的 Skill）
- 需要可观测性（每个节点的执行时间、输入输出）

### 为什么不用 ORM？

**直接使用 `pg` pool 的原因**：
- 查询逻辑简单，ORM 的抽象层带来的收益小
- 避免 ORM 的性能开销（N+1 查询问题）
- 更灵活的 SQL 控制（如 `ILIKE` 模糊匹配、未来的向量搜索）

### 为什么选择 PostgreSQL？

- 成熟稳定，Railway / Render 等平台原生支持
- 支持 JSON 字段（存储 `constraints`、`examples`）
- 未来扩展：pgvector 插件支持向量搜索

## 性能与可扩展性

### 当前性能指标

| 操作 | 延迟 | 瓶颈 |
|------|------|------|
| Agent 1 原子化一篇文章 | 3-5 秒 | LLM API 调用 |
| Agent 2 生成一篇文章 | 5-10 秒 | 知识检索 + LLM 生成 |
| Agent 3 生成一个 Skill | 3-5 秒 | LLM API 调用（3 次） |
| 知识库检索（关键词） | < 100ms | PostgreSQL ILIKE 查询 |

### 未来优化方向

**1. 向量语义搜索**
- 当前：关键词匹配（`ILIKE '%关键词%'`）
- 未来：向量搜索（`ORDER BY embedding <=> query_embedding`）
- 收益：召回率提升 30-50%（根据 RAG 领域经验）

**2. 卡片去重**
- 当前：无去重，相同观点可能存在多张卡片
- 未来：基于向量相似度去重（余弦相似度 > 0.9 视为重复）
- 收益：知识库体积减少 20-30%

**3. 增量索引**
- 当前：每次检索都扫描全表
- 未来：为 `content` 字段建立全文索引（GIN）
- 收益：检索延迟降低 50%

**4. 缓存热门卡片**
- 当前：每次都查数据库
- 未来：Redis 缓存高频引用的卡片
- 收益：检索延迟降低 70%（缓存命中时）

## 安全与隐私

### 数据隔离
- 所有表都有 `user_id` 字段
- 所有查询都带 `WHERE user_id = ?` 条件
- 用户 A 无法访问用户 B 的卡片或文章

### API 鉴权
- 所有 API 路由都使用 `requireAuth` 中间件
- Session 存储在 PostgreSQL（`connect-pg-simple`）
- 生产环境必须配置 `SESSION_SECRET`

### AI 调用安全
- API Key 存储在环境变量，不提交到代码仓库
- 支持任何 OpenAI 兼容的 API（用户可自建或使用第三方）
- 不存储 AI 返回的原始响应（仅存储结构化结果）

## 总结

AtomFlow 的核心创新在于：
1. **知识原子化**：将文章拆解为可复用的最小单元
2. **Agent 协作**：三个专职 Agent 分工明确，通过数据库协作
3. **风格定制**：自然语言描述风格，AI 自动生成结构化指令

这套架构的优势：
- **可扩展**：新增 Agent 不影响现有 Agent
- **可观测**：每个节点的执行轨迹可追踪
- **可维护**：每个 Agent 职责单一，代码清晰
